The problem

Most agent code becomes spaghetti by week two.

  • Tool dispatchers grow into giant if/elif chains.
  • Provider SDKs leak everywhere — switch from OpenAI to Groq and 40 files change.
  • Orchestrators hard-code the agent list. New agent = touch 6 files.
  • Tests mock the world because nothing is substitutable.
Why this matters02
What you'll leave with

A working mental model of LLM → Docker Swarm → parallel agents — and a SOLID layout you can copy.

  • Why an agent is just a tool — and a tool is just a JSON schema + a function.
  • Five SOLID principles applied to a real, runnable AI codebase.
  • How to swap a thread pool for a Docker Swarm executor in one commit.
  • How to test all of it without an LLM key or a Docker daemon.

Chapter 1

SOLID, applied to LLM systems.

Five old principles that keep agent codebases small, swappable, and testable — even when AI is in the loop.
S
Single Responsibility
Each agent does one thing. BaseAgent is one method.
O
Open / Closed
Add agents without touching the registry.
L
Liskov Substitution
Two executors, one Protocol. Swap freely.
I
Interface Segregation
LLMClient exposes one method: chat.
D
Dependency Inversion
Supervisor depends on ABCs, never on docker or openai.
SOLID06
Principle 01 · S
S
Single Responsibility
A class has one reason to change. For an agent, that reason is "the transform we apply to one input file."
# agents/base.py
class BaseAgent(ABC):
    name:        ClassVar[str]
    description: ClassVar[str]
    parameters:  ClassVar[dict]   # JSON Schema for the LLM

    @classmethod
    def tool_schema(cls) -> ToolSchema: ...

    @abstractmethod
    def run(self, input_path, output_dir, job_id) -> AgentResult: ...
Takeaway: the agent contract is intentionally tiny. Anything bigger is a different concern (logging, executing, scheduling) and lives elsewhere.
SOLID · S07
Principle 02 · O
O
Open for extension, closed for modification
Adding behaviour shouldn't require editing the orchestrator. A decorator + a module is the whole extension protocol.
# core/registry.py
class AgentRegistry:
    def register(self, cls):
        if cls.name in self._agents:
            raise AgentAlreadyRegisteredError(...)
        self._agents[cls.name] = cls
        return cls

REGISTRY = AgentRegistry()
def register_agent(cls): return REGISTRY.register(cls)
Zero name-based branching. Adding an agent never edits the registry, the supervisor, or the LLM client. The registry doesn't know what agents exist — only that they implement the contract.
SOLID · O08
Open/Closed in 15 lines

Adding a tool the LLM can call.

# agents/slogan_generator.py
from agents.base import AgentResult, BaseAgent
from core.registry import register_agent

@register_agent
class SloganGeneratorAgent(BaseAgent):
    name        = "slogan_generator"
    description = "Generate three catchy marketing slogans."
    parameters  = {"type": "object", "properties": {"input_ref": ...}}

    def __init__(self, llm=None):
        self._llm = llm

    def run(self, input_path, output_dir, job_id):
        text = input_path.read_text()
        body = self._llm.chat("You're a copywriter.", [{"role": "user", "content": text}], []).text
        out = output_dir / f"{job_id}__{self.name}.result"
        out.write_text(body)
        return AgentResult(self.name, job_id, out, summary="SLOGANS")
The supervisor, the LLM, the CLI, the Docker stack — none of them learn about this file. They discover it through the registry the next time they ask REGISTRY.names().
SOLID · O · live09
Principle 03 · L
L
Liskov Substitution
Anywhere the supervisor uses an executor, you can swap implementations and the supervisor doesn't notice.
# core/supervisor/supervisor.py
class AgentExecutor(Protocol):
    def execute(self, calls: list[tuple[str, str]]) -> list[AgentResult]: ...

class ThreadPoolAgentExecutor:   # dev / CI — fast, no Docker
    def execute(self, calls): ...

class SwarmAgentExecutor:        # prod / talk — real Swarm services
    def execute(self, calls): ...

Supervisor(executor=executor, ...)        # doesn't care which one
Why this matters for the talk: Epic 3's "swap to Swarm" was a one-commit diff because Supervisor only ever held an AgentExecutor, not a ThreadPoolAgentExecutor.
SOLID · L10
Principle 04 · I
I
Interface Segregation
Tiny interfaces. Fat ABCs force every consumer to know about methods they don't use.
# core/llm/base.py
class LLMClient(ABC):
    @abstractmethod
    def chat(
        self,
        system: str,
        messages: list[dict],
        tools: list[dict],
        tool_results: list[ToolResult] | None = None,
    ) -> LLMResponse: ...
One method. Streaming, embeddings, vision — those would be separate ABCs if we needed them. The supervisor never sees what it doesn't use. New providers reach parity in 10 LOC.
SOLID · I11
Principle 05 · D
D
Dependency Inversion
High-level modules don't depend on concretes. They depend on contracts. Concretes plug in at the composition root.
# core/supervisor/supervisor.py
class Supervisor:
    def __init__(self, *,
                 llm:      LLMClient,           # ABC, not OpenAI
                 registry: AgentRegistry,
                 executor: AgentExecutor,       # Protocol, not Swarm
                 volume:   SharedVolume):
        ...
Run grep "import docker\|import openai" core/supervisor/ — you get zero hits. The Supervisor never imports an SDK. Concrete wiring happens once, in supervisor.py (the CLI entrypoint).
SOLID · D12
Chapter 2

Where everything lives.

Module boundaries are how SOLID becomes real instead of aspirational.
supervisor.py             # composition root (CLI)
agents/
  base.py                 # BaseAgent + ToolSchema
  runner.py               # container entrypoint
  capitalize.py · reverse.py · ...
core/
  registry.py             # Open/Closed extension point
  llm/                    # LLMClient ABC + 3 adapters
  swarm/                  # SwarmManager + ResultWatcher
  supervisor/             # Supervisor + executors
  io/shared_volume.py     # file-handoff primitive
  logging.py              # rich event logger
docker/                   # Dockerfile + stack files
scripts/demo-*.sh         # 4-command demo flow
tests/{unit, integration}
  • core/ is the only place SDKs are imported. Lazy imports keep it lean.
  • agents/ depends on core but core never imports an agent — the registry inverts that.
  • docker/ is configuration, not code. It's the deployment shape, not the design.
  • scripts/ is what the audience runs on stage. Four commands, zero magic.
  • tests/ is split: unit needs nothing, integration opts in with one env var.

Pattern · Provider-agnostic LLM

Three providers in <30 LOC of subclasses.

# core/llm/openai_compat.py — shared base, ~50 LOC
class _OpenAICompatibleClient(LLMClient):
    BASE_URL:    ClassVar[str | None] = None
    API_KEY_ENV: ClassVar[str] = ""

    def __init__(self, model=None):
        api_key = os.environ[self.API_KEY_ENV]
        self._client = OpenAI(api_key=api_key, base_url=self.BASE_URL)

# Three providers — only URL + env var differ.
class OpenAIClient(_OpenAICompatibleClient):
    API_KEY_ENV = "OPENAI_API_KEY";  DEFAULT_MODEL = "gpt-4o-mini"

class GroqClient(_OpenAICompatibleClient):
    BASE_URL = "https://api.groq.com/openai/v1"
    API_KEY_ENV = "GROQ_API_KEY";    DEFAULT_MODEL = "llama-3.3-70b-versatile"

class XAIClient(_OpenAICompatibleClient):
    BASE_URL = "https://api.x.ai/v1"
    API_KEY_ENV = "XAI_API_KEY";     DEFAULT_MODEL = "grok-2-latest"
Switching provider on stage: one env var. Zero code changes. Because Groq and xAI are OpenAI-API-compatible, the SDK does the work — we just point base_url at them.
Pattern · LLM14
Subtle but important

tool_schema() is a classmethod. Here's why.

# So we can collect schemas without instantiating.
schemas = [
    cls.tool_schema()
    for cls in REGISTRY.classes()
]

# If it weren't a classmethod, we'd have to:
schemas = [
    cls(llm=get_llm_client()).tool_schema()
    for cls in REGISTRY.classes()
]
# …and every agent would need an LLM key
#    just to advertise its name to the model.

LLM-aware agents need real dependencies at __init__ — an LLMClient, an HTTP session, a credential. Construction has side effects.

Schemas are inert metadata. The registry should read them without paying construction cost.

Lesson: when something is data about a class, make it accessible from the class — not from an instance.

The supervisor loop.

Every LLM agent system you'll build comes down to this loop. Internalize it once.
def run(self, prompt: str, *, job_id=None) -> str:
    job_id = job_id or _new_id()
    self._volume.write_input(job_id, prompt)

    tools    = self._registry.openai_tools()
    messages = [{"role": "user", "content": build_user_message(prompt, job_id)}]

    for i in range(self._max_iterations):
        resp = self._llm.chat(SYSTEM_PROMPT, messages, tools=tools)

        if resp.is_final:
            return resp.text or ""

        messages.append(build_assistant_tool_call_message(resp))
        calls   = [(tc.name, job_id) for tc in resp.tool_calls]
        results = self._executor.execute(calls)          # parallel
        messages.extend(build_tool_result_messages(resp.tool_calls, results))

    raise SupervisorIterationLimitError(...)

Parallel fan-out17
Why Docker Swarm

The "lazy K8s" — and that's the point.

  • Single-node demo: one command, docker swarm init. No control plane to operate.
  • Same primitive scales: add nodes with swarm join. Production-grade ergonomics.
  • Python SDK is enough: docker.from_env().services.create(...). ~150 LOC of SwarmManager.
  • Ephemeral services: spawn → run-to-completion → cleanup. Same shape as a Lambda invocation.
  • Audience aha-moment: agents appear in docker service ls in real time. K8s hides that.

The substitution payoff

Threadpool to Swarm in one commit.

We built Epic 2 entirely with a ThreadPoolAgentExecutor. The LLM loop ran end-to-end without any Docker involvement.

Epic 3 introduced SwarmAgentExecutor implementing the same execute(calls) contract.

The Supervisor class — the most important file in the codebase — did not change.

The litmus test for a SOLID design: a change that "should" be small actually is small.
# supervisor.py CLI — the only difference
def _build_executor(kind, volume, llm, logger):
    if kind == "threadpool":
        return ThreadPoolAgentExecutor(
            REGISTRY, volume, llm=llm
        )
    if kind == "swarm":
        swarm   = SwarmManager(image="agent-swarm:latest", logger=logger)
        watcher = ResultWatcher(volume.results_dir)
        return SwarmAgentExecutor(
            REGISTRY, volume, swarm, watcher, logger=logger
        )

Test doubles fall out for free

Protocol-typed deps make fakes trivial.

# tests/conftest.py
class FakeLLMClient(LLMClient):
    def __init__(self):
        self._responses = []
        self.calls = []

    def queue_response(self, r):
        self._responses.append(r)

    def chat(self, system, messages, tools, tool_results=None):
        self.calls.append({"system": system, ...})
        return self._responses.pop(0)

# tests/unit/test_supervisor.py
def test_two_parallel_tools_then_final(volume, executor, fake_llm):
    fake_llm.queue_response(LLMResponse(
        text=None,
        tool_calls=(ToolCall("c1", "capitalize", {...}),
                    ToolCall("c2", "reverse", {...}))
    ))
    fake_llm.queue_response(LLMResponse(text="Final report."))

    result = Supervisor(llm=fake_llm, ...).run("Process...", job_id="j")
    assert "Final report" in result

Delivery22
When you go back to your codebase

A 5-question SOLID checklist.

  1. S: Can I describe each class with one sentence — without using "and"? If not, split it.
  2. O: What's the friction cost of adding a new variant — a tool, a provider, an agent? One file or many?
  3. L: Do my consumer types name a concrete class, or an ABC/Protocol? Concrete = coupling.
  4. I: Are there abstract methods my consumers never call? Split the ABC.
  5. D: Where do concrete deps get wired? If everywhere, you have no composition root.
If three of these fail, you'll spend more time renaming files than building features. Refactor before adding the next agent.
Checklist23
The obvious question

Why not LangChain / Semantic Kernel / CrewAI?

  • Tool-use is now a vendor primitive. OpenAI/Groq/xAI all serve the same function-calling format. The abstraction we needed was already there.
  • The supervisor loop is ~30 LOC. Frameworks add 50KB of code to wrap something we can read in two minutes.
  • You can't substitute what you can't see. Heavy frameworks bury the executor behind config — exactly where SOLID matters most.
  • Use frameworks for what they're good at: retrieval pipelines, document loaders, chain-of-thought scaffolds. Not "agent as service."
The cost is wrong. When the whole abstraction fits on one slide, you don't import a framework to replace it — you write it once and own it.
Frameworks24
Things we hit on the way

Real-world friction worth flagging.

  • Docker socket mount in the supervisor container is root-equivalent. Fine for demo, smell in production. Use a remote Docker API with mTLS instead.
  • LLM-aware agents inside a ScriptedLLMClient recursively consume responses. Our talk fixture deliberately omits them — and the test catches the failure mode.
  • Workers in the original codebase ran a glob-once-and-exit loop. The race meant the demo never actually worked. Caught it during regression testing.
  • Stage WiFi melts. --dry-run --fixture is the answer; commit it before you need it.
  • shared-data volume is root-owned from inside Docker. Cleanup needs a sidecar alpine rm.

Three things to remember

If you walk away with three ideas:

  1. Agents are tools. Tools are JSON schemas plus a function. Don't let frameworks talk you out of how simple this is.
  2. Two implementations of one abstraction beats one good implementation. The second one forces the abstraction to be right.
  3. Ship by epic, not by feature. Demo-readiness gates make refactors honest — every checkpoint has to work on stage.
